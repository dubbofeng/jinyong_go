# 数据管理策略文档

## 概述

本项目采用 **混合数据源策略**，结合 JSON 文件和数据库的优势：
- **JSON 文件**：版本控制、快速开发、静态配置
- **数据库**：动态管理、后台编辑、用户数据

---

## 数据分类

### 📁 类型一：纯 JSON 文件（不入库）

**特点**：保持在代码仓库中，随版本发布，不支持运行时修改

#### 1. 对话数据
- **位置**：`src/data/dialogues/`
- **格式**：`{npcId}.{locale}.json`
- **用途**：NPC 对话树、剧情文本
- **原因**：
  - ✅ 内容量大，适合文本编辑器
  - ✅ 多语言管理方便（独立文件）
  - ✅ Git diff 清晰可见变更
  - ✅ 剧情固定，无需运行时修改

**示例**：
```
dialogues/
  ├── hong_qigong.zh.json
  ├── hong_qigong.en.json
  ├── linghu_chong.zh.json
  └── linghu_chong.en.json
```

#### 2. Tilemap 地图配置
- **位置**：`src/data/maps/`
- **格式**：`{mapId}.json`
- **用途**：传统正交视图地图的静态配置
- **原因**：
  - ✅ 地图数据结构复杂（多层、瓦片数组）
  - ✅ 作为地图模板，不需要动态修改
  - ✅ 可能被 Isometric 地图完全替换

**示例**：
```json
{
  "id": "huashan",
  "name": "华山传功厅",
  "width": 25,
  "height": 18,
  "layers": [...]
}
```

#### 3. AI 提示词模板
- **位置**：`src/lib/image-prompts.ts`
- **格式**：TypeScript 代码
- **用途**：AI 图片生成的提示词配置
- **原因**：
  - ✅ 配置较少，代码管理即可
  - ✅ TypeScript 类型检查
  - ✅ 无需频繁修改

---

### 🔄 类型二：JSON 种子 + 数据库运行时

**特点**：JSON 作为初始配置，首次加载到数据库，后续通过数据库/后台管理

#### 1. NPC 配置
- **种子文件**：`src/data/npcs.json`
- **数据库表**：`npcs`
- **加载脚本**：`scripts/seed-data.ts`

**工作流程**：
```
1. 开发：编辑 src/data/npcs.json
2. 部署：运行 npx tsx scripts/seed-data.ts
3. 运行：从数据库读取 NPC 数据
4. 管理：后台界面修改 NPC 位置、属性
5. 重置：删除数据库记录，重新运行种子脚本
```

**数据结构**：
```json
{
  "npcId": "hong_qigong",
  "name": "洪七公",
  "mapId": "huashan",
  "positionX": 12,
  "positionY": 9,
  "npcType": "teacher",
  "teachableSkills": ["kanglongyouhui"]
}
```

**为什么混合**：
- ✅ JSON 方便批量初始化和版本控制
- ✅ 数据库支持后台动态调整 NPC 位置
- ✅ 可以根据游戏进度动态显示/隐藏 NPC

#### 2. 任务配置
- **种子文件**：`src/data/quests.json`
- **数据库表**：`quests`
- **加载脚本**：`scripts/seed-data.ts`

**工作流程**：同 NPC 配置

**数据结构**：
```json
{
  "questId": "first_battle",
  "title": "初试锋芒",
  "chapter": 1,
  "requirements": {...},
  "rewards": {...}
}
```

**为什么混合**：
- ✅ 任务内容相对固定，适合 JSON 编辑
- ✅ 支持运营期间添加活动任务（数据库插入）
- ✅ 可以动态调整任务奖励（平衡性调整）

---

### 💾 类型三：纯数据库

**特点**：完全由运行时生成，不需要初始配置文件

#### 1. Isometric 地图系统
- **数据库表**：`maps`, `mapTiles`, `mapItems`
- **创建方式**：
  - 后台地图生成器
  - 地图编辑器
  - API 接口

**为什么纯数据库**：
- ✅ 地图由算法生成（Perlin Noise）
- ✅ 支持后台可视化编辑
- ✅ 地图物品可动态添加/删除
- ✅ 支持用户创建自定义地图（未来）

#### 2. 用户数据
- **数据库表**：`users`, `gameProgress`
- **特点**：运行时动态生成

#### 3. 对战记录
- **数据库表**：`chessRecords`
- **特点**：每场游戏自动记录

## 数据加载优先级

### 读取策略

**NPC 数据**：
```typescript
// 1. 尝试从数据库读取
const dbNPC = await db.select().from(npcs).where(eq(npcs.npcId, id));

// 2. 如果不存在，从 JSON 读取
if (!dbNPC) {
  const jsonNPC = loadFromJSON('npcs.json', id);
}
```

**对话数据**：
```typescript
// 直接从 JSON 读取（不经过数据库）
const dialogue = loadDialogue(npcId, locale);
```

**地图数据**：
```typescript
// Tilemap：从 JSON 读取
const tilemapConfig = loadTilemapConfig(mapId);

// Isometric：从数据库读取
const isometricMap = await loadIsometricMap(mapId);
```

## 使用指南

### 开发环境

#### 1. 首次设置
```bash
# 运行数据库迁移
pnpm drizzle-kit push

# 加载种子数据
npx tsx scripts/seed-data.ts

# 初始化示例地图
npx tsx scripts/init-maps.ts
```

#### 2. 修改静态配置（对话、任务）
```bash
# 1. 编辑 JSON 文件
vim src/data/dialogues/hong_qigong.zh.json

# 2. 刷新页面即可看到效果（开发模式热更新）
```

#### 3. 修改 NPC/任务配置
```bash
# 方式一：修改 JSON + 重新加载
vim src/data/npcs.json
npx tsx scripts/seed-data.ts

# 方式二：通过后台管理界面修改（推荐）
# 访问 http://localhost:3000/admin
```

#### 4. 重置数据到初始状态
```bash
# 删除并重新创建数据库表
pnpm drizzle-kit push

# 重新加载种子数据
npx tsx scripts/seed-data.ts
npx tsx scripts/init-maps.ts
```

### 生产环境

#### 1. 部署流程
```bash
# 1. 构建应用
pnpm build

# 2. 运行数据库迁移
pnpm drizzle-kit push

# 3. 加载种子数据（仅首次部署）
npx tsx scripts/seed-data.ts
npx tsx scripts/init-maps.ts

# 4. 启动应用
pnpm start
```

#### 2. 更新内容
```bash
# 静态内容（对话）：随代码部署自动更新
git push && vercel deploy

# 动态内容（NPC位置）：通过后台管理界面修改
# 无需重新部署
```

## API 使用示例

### 在组件中加载数据

```typescript
import { loadNPC, loadDialogue, loadAllQuests } from '@/src/lib/data-loader';

// 加载单个 NPC（优先数据库）
const npc = await loadNPC('hong_qigong');

// 加载对话（直接从 JSON）
const dialogue = loadDialogue('hong_qigong', 'zh');

// 加载所有任务（优先数据库）
const quests = await loadAllQuests();
```

### 在 API 路由中使用

```typescript
// app/api/npcs/[npcId]/route.ts
import { loadNPC } from '@/src/lib/data-loader';

export async function GET(
  request: Request,
  { params }: { params: { npcId: string } }
) {
  const npc = await loadNPC(params.npcId);
  return Response.json(npc);
}
```

## 最佳实践

### ✅ 推荐做法

1. **剧情对话**：保持在 JSON 文件中
   - 便于作家编辑
   - Git 版本控制
   - 多语言文件独立

2. **NPC 位置**：使用数据库
   - 后台可视化编辑
   - 支持动态调整

3. **地图生成**：使用数据库
   - 算法生成后保存
   - 支持编辑器修改

4. **用户数据**：始终使用数据库
   - 唯一可靠的存储方式

### ❌ 避免做法

1. 不要把大量静态文本放入数据库
   - 增加数据库负担
   - 失去版本控制
   - 编辑不便

2. 不要把用户数据放入 JSON
   - 无法扩展
   - 数据不安全

3. 不要在没有备份的情况下修改生产数据库
   - 使用后台管理界面
   - 启用操作日志

## 故障排除

### 问题：种子数据加载失败
```bash
# 检查数据库连接
psql -U postgres -d jinyong_go

# 检查表结构
pnpm drizzle-kit push

# 重新加载
npx tsx scripts/seed-data.ts
```

### 问题：对话数据不显示
```bash
# 检查文件路径
ls src/data/dialogues/

# 检查 JSON 格式
npx prettier --check src/data/dialogues/*.json
```

### 问题：NPC 位置不正确
```bash
# 重新加载种子数据（会覆盖数据库中的修改）
npx tsx scripts/seed-data.ts

# 或者通过后台管理界面手动调整
```

## 总结

| 数据类型 | 存储位置 | 版本控制 | 运行时修改 | 使用场景 |
|---------|---------|---------|-----------|---------|
| 对话数据 | JSON | ✅ | ❌ | 剧情文本 |
| Tilemap | JSON | ✅ | ❌ | 地图模板 |
| NPC配置 | JSON→DB | ✅ | ✅ | 初始配置+动态调整 |
| 任务配置 | JSON→DB | ✅ | ✅ | 初始配置+运营活动 |
| Isometric地图 | DB | ❌ | ✅ | 动态生成和编辑 |
| 用户数据 | DB | ❌ | ✅ | 玩家进度 |
| 对战记录 | DB | ❌ | ✅ | 游戏记录 |

这个混合策略充分利用了两种存储方式的优势，既保持了开发灵活性，又支持了运营需求。
