## 二、迁移方案设计

### 方案A：渐进式迁移（推荐）

**优势**:
- ✅ 风险低，可逐步验证功能
- ✅ Tilemap系统保留，向后兼容
- ✅ 可同时开发和测试两套系统
- ✅ 用户体验平滑过渡

**劣势**:
- ⚠️ 维护成本高（同时维护两套系统）
- ⚠️ 代码复杂度增加

**实施步骤**:

#### 阶段1：Isometric渲染引擎开发（4.2核心功能）
1. 创建 `src/lib/isometric-engine.ts`
   - Canvas渲染系统
   - 瓦片排序（z-index）
   - 等距投影计算
   - 资源加载器

2. 创建 `src/components/IsometricGame.tsx`
   - 独立的Isometric游戏组件
   - 游戏循环和输入处理
   - 与现有RPGGame并行存在

#### 阶段2：玩家移动系统（4.2）
1. 点击瓦片移动
2. A* 路径寻找算法
3. 平滑移动动画（deltaTime）
4. 碰撞检测（地形、建筑）

#### 阶段3：NPC与传送门集成（4.2）
1. NPC作为MapItems存储
2. NPC精灵图渲染
3. 点击交互触发对话
4. 传送门视觉效果和交互

#### 阶段4：地图内容创建（4.3）
1. 世界地图（64×64）
2. 3个场景地图（32×32）- 对应Tilemap的3个地图
3. 使用地图编辑器放置NPC、传送门

#### 阶段5：主游戏入口切换（4.4）
1. 在 `app/[locale]/game/page.tsx` 添加地图类型切换
2. 用户可选择 Tilemap 或 Isometric
3. 数据迁移工具（Tilemap JSON → Isometric DB）

#### 阶段6：完全迁移（4.4）
1. 默认使用Isometric
2. 保留Tilemap引擎代码（但不再使用）
3. 完整测试所有功能

**时间估算**: 5-7天

---

### 方案B：一次性替换（高风险）

**优势**:
- ✅ 代码简洁，无冗余
- ✅ 维护成本低

**劣势**:
- ❌ 风险高，所有功能必须一次性完成
- ❌ 测试周期长
- ❌ 无回退方案

**不推荐理由**:
- MVP项目需要快速迭代
- Tilemap系统已稳定运行，有NPC对话和传送门
- Isometric系统尚未验证前端性能

## 四、技术风险评估

### 4.1 性能风险

**Isometric渲染复杂度**:
- ⚠️ 需要瓦片深度排序（z-index）
- ⚠️ 建筑精灵图尺寸较大（可能影响加载速度）
- ⚠️ 64×64世界地图 = 4096个瓦片（需要优化）

**缓解措施**:
- 实现视口裁剪（只渲染可见区域）
- 离屏Canvas缓存（类似Tilemap）
- 精灵图懒加载
- WebGL渲染优化（可选）

### 4.2 坐标转换风险

**问题**:
- Tilemap使用正交坐标（x, y）
- Isometric使用等距坐标（screenX, screenY）
- 碰撞检测、路径寻找需要坐标转换

**缓解措施**:
- 使用 `src/lib/map/isometricUtils.ts` 工具函数
- 单元测试覆盖坐标转换
- A* 算法在笛卡尔坐标系计算，渲染时转换

### 4.3 数据迁移风险

**问题**:
- 现有3个Tilemap地图（JSON文件）
- 需要转换为Isometric数据库格式

**缓解措施**:
- 创建迁移脚本 `scripts/migrate-tilemap-to-isometric.ts`
- 保留原始JSON文件（不删除）
- 手动验证迁移数据的准确性

## 六、关键决策点

### 决策1：是否保留Tilemap系统？
**推荐**: ✅ 保留代码，但不再使用
- 理由：向后兼容，降低风险
- 代码位置：`src/lib/tilemap-engine.ts` 标记为 `@deprecated`

### 决策2：Isometric资源生成优先级
**推荐**: 优先生成NPC精灵图和传送门特效
- 理由：这些是功能关键资源
- 使用AI生成（现有美术管理系统）

### 决策3：A* 路径寻找算法库
**推荐**: 使用 `pathfinding` npm包
```bash
pnpm add pathfinding
pnpm add -D @types/pathfinding
```
- 理由：成熟、稳定、性能好

### 决策4：Isometric渲染库
**推荐**: 纯Canvas实现（不使用PixiJS/Phaser）
- 理由：轻量级，更灵活，与现有架构一致
